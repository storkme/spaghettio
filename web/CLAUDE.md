# Web app — rendering rules

Scoped notes for `web/`. Read alongside the root `CLAUDE.md`; this file
exists because these constraints are invisible from the code you're
likely editing and have already cost a full round-trip (quality badges,
2026-07-20 — implemented in the wrong renderer, then nearly baked into
shared textures).

## There are TWO renderers — features must land in both (or be knowingly scoped)

| Path | Entry points | Used by |
|---|---|---|
| **Particle path** (what the user sees) | `renderer/particleLayout.ts` (`commitEntityAsParticle`, `renderLayoutAsParticles`) + `renderer/streamingRenderer.ts` | The live generator view — every sidebar solve renders here |
| **Graphics path** | `renderer/entities.ts` (`renderLayout`, `drawEntityGraphic`) | Corpus browser, parsed community blueprints, snapshot loader |

A visual change made only in `entities.ts` will look done in corpus
views and be **completely absent from the live generator**. When you
add an entity-level visual, grep for how the same concept is handled in
both paths (e.g. carries icons: `iconParticle` in `particleLayout.ts`
vs the icon drawing in `entities.ts`). Design + perf background:
`docs/rfp-renderer-particle-container.md`,
`docs/web-render-perf-investigation.md`.

## Particle-path atlas textures are SHARED per entity type

`getEntityAtlasTexture` bakes one texture per key —
`machineAtlasKey(name)`, `inserterAtlasKey(name, dir)`, etc. — and
every instance of that entity type reuses it. Two hard rules follow:

- **Never bake per-entity state into an atlas texture.** Whichever
  instance renders first would stamp its state onto every clone (the
  quality-badge near-miss). `getEntityAtlasTexture` strips
  `entity.quality` at the top for exactly this reason; extend that
  strip if `PlacedEntity` grows more per-entity visual fields.
- **Per-entity visuals are separate particles** in the icon layer,
  keyed into `particleMap` alongside the entity. Follow the
  `entry.icon` / `entry.badge` pattern: the new particle must track the
  entity's alpha at EVERY site that touches `entry.icon` — the
  highlight animator (`animateTo`, tick, cancelAll), reveal fades
  (`applyParticleReveals`), scrub mode (`getParticleReveals` →
  `streamingRenderer`'s reveal list), and the legacy controller. Miss
  one and the overlay ghosts at full alpha while its entity dims.

## Textures must be preloaded before first commit

Committed particles never pick up textures that arrive later. Anything
drawn at commit time needs its texture in the Pixi asset cache first:
machine frames + badge icons load in `main.ts` init
(`initEntityIcons`, `preloadQualityBadgeIcons`); carries icons preload
per-layout in the sidebar solve flow before streaming starts. A new
texture-consuming feature needs a preload call at the right layer — or
it renders as the fallback/placeholder forever.

## Misc

- Item/entity icon PNGs in `public/icons/` are extracted from the game
  by `scripts/extract_icons.py` (includes the `__quality__` mod's tier
  badges). Don't hand-edit; re-run the extractor.
- `src/wasm-pkg/` is gitignored and regenerated by `npm run dev` / CI;
  after Rust changes ask the user to restart `npm run dev` rather than
  invoking wasm-pack ad hoc (root CLAUDE.md has the manual command).
- Visual polish is validated by the user, not by screenshot iteration —
  make the change, hand over a URL.
