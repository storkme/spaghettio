### Attempted and reverted — "merge adjacent trunk+tap into one `trunktap:` spec"

The plan is still in `/home/stork/.claude/plans/abundant-gliding-turing.md`.
Phase 1 (row metadata) is done and shipped. Phase 2 (structural
`ConnectiveStep` refactor) is **not started** — the "merge" experiment was a
much smaller hack that tried to get the Phase 1 fix over the line without the
full Phase 2 lift.

The merge idea: emit `trunk:copper-cable:4:10..46` and the adjacent
`tap:copper-cable:...` as a single L-shaped `trunktap:` spec so
`classify_crossing` / `to_junction` see real path deltas and can derive flow
direction, and region-growth can promote encountered specs into the SAT
zone.

**What worked:**

- `tier2_electronic_circuit` moved from `belt-item-isolation` → back to its
  original single sideload warning. The merge itself fired, the trunktap got
  stamped, SAT saw the right specs.
- Region-growth promotion (encountered → participating) worked cleanly.

**What broke:** `tier2_electronic_circuit_20s_from_ore` (previously green)
picked up 6–11 `belt-dead-end` errors. Root cause is **not** the merge itself;
it's four separate cascading issues in the downstream pipeline that each
needed independent fixes:

1. **Materialization ordering.** `entities.retain(… !ghost_item_at.contains_key(&tile))`
   silently drops the second spec's entity at any crossing. Fine when the
   "second spec" is a short tap that will be re-added by corridor/per-tile
   template. Deadly when it's the interior of a 36-tile trunktap — the
   interior tile gets erased and nothing re-adds it.
2. **Corridor endpoint check.** Uses `is_permanent`; needs to allow the
   bridged spec's own tiles. Swap for the `unreleasable_obstacles` set I
   added in `76af036`. Easy fix.
3. **Corridor retain-and-readd.** The `entities.retain` that drops
   `ghost:{key}` doesn't match `trunk:{item}` entities from the merged spec;
   adding a dual match fixes it in the obvious place, but leaves
   stale-claim issues with unrelated trunks whose tiles sit inside the
   corridor zone.
4. **Region-growth promotion interaction with multi-crossing cases.**
   Corridor runs at y=52/61/69 in `20s_from_ore` each want their own SAT
   view of the world, and promoting a merged trunktap into one zone changed
   the forbidden set for the next zone's solve.

After several iterations I ran out of budget to converge and `git restore`d
everything Phase-5/6. Baseline is clean.

### The "right shape" in retrospect

Don't merge at the `BeltSpec` level. Instead, do Phase 2 of the plan:

1. Introduce `ConnectiveStep` with constructors that make degenerate
   (length-1, wrong-direction, empty-range) specs *unconstructable*.
2. Rewrite the emission layer in `ghost_router.rs:260-436` to build
   `Vec<ConnectiveStep>`, not `Vec<BeltSpec>`.
3. Lower each variant in `connective_lowering.rs`. `TrunkRun` becomes a
   direct stamp (no A*). `ProducerBend` becomes a one-tile direct stamp.
   Only `TapEast` / `ReturnRun` / `FamilyFeeder` hit A*.
4. Fix materialization to **preserve both specs at crossings**, marking the
   tile as dual-claim in occupancy instead of silently dropping one.
   Corridor / per-tile templates expect dual-claim tiles inside their
   zones.
5. Only after (4) is solid, add region-growth promotion.

Estimated size: ~600 LOC, 3–5 commits, needs a dedicated session. The hack
route (~200 LOC one-shot) is not viable — it cascades through corridor
handling, materialization, and promotion semantics in ways that need careful
untangling.

## Concerns / exploration areas

### 1. `classify_crossing` silently drops 1-tile specs

At `ghost_router.rs:1585-1593` (around `classify_crossing` / `to_junction`),
specs with a single-tile path have no flow direction to derive from deltas,
so they're dropped from `crossing_specs`. When that leaves a crossing with
`<2` specs it falls through to `remaining_crossings` instead of reaching SAT
— which is how the unresolved region at `(4,10)` in `tier2_electronic_circuit`
comes about.

**Fix shape:** derive direction from `BeltSpec.exit_dir` (and a symmetric
`entry_dir` if needed) rather than from path deltas. Small change once
`entry_dir` exists; meaningful once 1-tile specs stop being degenerate.

### 2. `LayoutRegion.ports` is always empty

`ghost_router.rs:1239` pushes every region with `ports: Vec::new()`. The
`regionOverlay.ts` work I did this session (dashed input/output connectors,
port markers, IN/OUT labels) is entirely inert in the browser because there
is nothing to render. Populating this is the blocker for the "make the
input/output visualization clearer" task the user asked for.

**Fix shape:** at the region-collection site, read from the spec frontier
(inputs) and the growth-loop exit tiles (outputs) and emit `RegionPort`
entries. The data all exists in `JunctionStrategyContext`; it's just never
copied out.

### 3. The `SatStrategy` is live, not a placeholder

Noted here because I spent time last session thinking it was inert based on a
stale comment. PR #145 fixed it. `junction_sat_strategy.rs` waits for
`ctx.region.tile_count() > 1`, builds a `CrossingZone`, calls
`solve_crossing_zone`. Anything claiming "SAT is a placeholder" in a comment
or RFP should be deleted on sight.

### 4. `BeltSpec.entry_dir` is the missing twin of `exit_dir`

`exit_dir` was added in `66afe77` to let the producer side encode "items
leave heading West". The symmetric "items arrive from the North" case
(splitter body tile dropping south into a tap) has no encoding yet — that's
why the iron-plate input at `(2,10)` was mis-explained last session as
coming from the east. If `entry_dir` existed, the junction solver could
distinguish "sideload from above" (legal) from "sideload from the east"
(illegal into a UG input).

### 5. Tier3 / tier4 ghost-mode blockers

All tier3+ tests are currently `#[ignore = "ghost-mode: ..."]`. The shared
theme is that the ghost router can't yet handle feeder paths on fluid rows
(#68) or the balancer shapes on `advanced-circuit` (#136). These aren't
blocked on the trunktap merge; they're their own piece of work.

### 6. Corridor template retain logic is fragile

The pattern of "emit ghosts → corridor/perp template retains-and-readds a
subset" is load-bearing and brittle. Every time I add a new segment-id
prefix or change how one is stamped, the `retain` predicate in
`ghost_router.rs` needs to be updated in lockstep and there is no type-level
signal telling me which `retain` calls care. A small refactor to
*centralize* the "which seg-ids survive each phase" decision into one helper
would prevent a whole class of future bugs.
