# Fucktorio Codebase Audit — 2026-04-24

## Executive Summary

Fucktorio is a **mature, well-architected** automated Factorio factory blueprint generator. The project has evolved from a Python spaghetti-router to a Rust-based deterministic bus-layout engine with ghost-A* routing, SAT-backed junction solving, and a rich interactive web UI. All 9 e2e tests pass (tier 1-4), the codebase is ~15k lines of production Rust, and the deployment pipeline (GitHub Pages) is operational.

The codebase is in **good structural health** - documentation is thorough, contracts are explicit, and the validation framework provides real safety. However, several areas carry meaningful technical debt, and the project's roadmap has several open gaps (tier 5+ recipes, fluid row pitch, balancer template coverage, SAT zone editor polish).

---

## 1. Architecture Assessment

### 1.1 Pipeline Overview

```
Solver (recipe resolution)
  -> Bus Layout (row placement, lane planning, ghost routing)
    -> Blueprint Export (JSON + zlib + base64)
      -> Validation (23 checks)
```

**Single-pass, no retry loops.** The layout engine commits to a deterministic plan: estimate bus width -> place rows -> plan lanes -> compute gaps -> place rows (pass 2) -> place poles -> ghost route -> output mergers. This is a strength: the pipeline is predictable and debuggable.

### 1.2 Module Structure

| Crate | Lines | Responsibility | Health |
|-------|-------|---------------|--------|
| `fucktorio_core` | ~12k | Pure shared logic (solver, layout, export, validation) | Excellent |
| `fucktorio_wasm` | ~200 | Thin wasm-bindgen wrapper | Clean |
| `fucktorio_mining` | ~300 | CLI blueprint analyzer | Useful |
| `web` (TS) | ~12k | PixiJS UI + trace overlays | Growing complex |

### 1.3 Key Strengths

1. **Ghost pipeline contracts** (`docs/ghost-pipeline-contracts.md`) - The most valuable document in the repo. Every phase has explicit consume/produce/promises/does-not-promises invariants. This is the kind of documentation that prevents entire classes of bugs.

2. **Typed occupancy map** (`ghost_occupancy.rs`) - The `Occupancy` enum (`HardObstacle`, `RowEntity`, `Permanent`, `GhostSurface`, `Template`, `SatSolved`) provides a clear, type-safe model for tile ownership. This is the single most important data structure in the layout engine.

3. **Structured tracing** (`trace.rs`) - Every pipeline phase emits `TraceEvent` variants that feed the debug UI, snapshot debugger, and diagnostic tests. The trace system is a genuine engineering asset, not an afterthought.

4. **23-check validation framework** - Covers belt flow, structural, fluids, inserters, power, and underground checks. Parallelized via rayon. The validator is a safety net that catches regressions.

5. **E2E test harness with scoreboards** - 9 non-ignored tests covering tier 1-4, plus ignored diagnostics for ghost-cluster analysis and SAT zone histograms. Tests are comprehensive and well-structured.

6. **Deterministic, cross-platform** - Uses `BTreeMap` for iteration order, `FxHashMap` for lookup, and `indexmap` for recipe insertion order. The solver produces identical results on native and WASM targets.

## 2. Detailed Findings by Module

### 2.1 Solver (`solver.rs`, 242 lines) - Solid

- Recursive recipe resolution with cycle detection (`resolving` set)
- Handles probability-based products (e.g., mining drills)
- Supports recipe exclusion for steering (e.g., avoiding oil chain for plastic-bar)
- **Gap**: No support for alternative recipes with different energy requirements or throughput characteristics. The solver picks the first matching recipe from `recipes.json` without comparing efficiency.

### 2.2 Recipe Database (`recipe_db.rs`, 297 lines) - Good

- Bundled `recipes.json` embedded via `include_str!`
- Excludes recycling/crushing categories and barrel-emptying recipes
- Machine crafting speeds loaded from JSON
- **Gap**: The recipe selection is deterministic-first-match. There's no recipe ranking or multi-recipe exploration. For complex recipes (tier 4+), this means the solver may pick suboptimal recipes.

### 2.3 Bus Layout Engine - Excellent

#### `layout.rs` (750 lines)
- Two-pass row placement with gap computation
- Pole placement with wire-reach union-find
- Inter-row band detection
- **Gap**: `repair_pole_connectivity` (20 iterations, Chebyshev distance) is a heuristic that may fail on very wide layouts. No fallback to placing additional poles in strategic locations.

#### `lane_planner.rs` (948 lines)
- Bus lane planning with capacity-based tier selection
- Lane order optimization (exact search <=7 lanes, hill-climb above)
- `LaneFamily` creation for N->M balancer blocks
- **Gap**: Lane capacity table uses fixed per-lane values (7.5/15/22.5). These assume perfect lane utilization. In practice, inserters and machine throughput create uneven distribution. The validator catches this as `lane-throughput` warnings, but the planner doesn't account for it.

#### `ghost_router.rs` (3,288 lines) - Excellent
- 7-phase ghost routing: obstacle build -> splitter stamps -> balancer stamps -> spec emission -> negotiated A* -> materialization -> junction solving -> output mergers
- Negotiated congestion with history + present penalties
- Per-tile junction solver with strategy stack (perpendicular template + SAT fallback)
- Typed `Occupancy` map integration
- **Gap**: The negotiation loop has hardcoded limits (`MAX_NEGOTIATION_ITERATIONS = 8`, `MAX_NO_IMPROVEMENT = 2`). For very dense layouts, these may be insufficient. The `TURN_PENALTY = 8` is also hardcoded.

#### `junction_solver.rs` + `junction_sat_strategy.rs`
- Region-growth junction solver with strategy trait
- SAT-backed crossing zone solving (varisat)
- **Gap**: Tier 2/3 strategies (veto-directed growth, band regions) are partially implemented but not fully integrated. The strategy stack is `[PerpendicularTemplateStrategy, SatStrategy]` - more strategies are defined but not all are active.

### 2.4 Validation Framework - Good

#### `belt_flow.rs` (3,789 lines) - Largest module
- Belt connectivity, flow paths, reachability, network topology, junctions
- Underground belt pair matching and sideloading checks
- Input rate delivery verification
- **Gap**: Belt flow validation uses a Kahn topo-sort with splitter pairing and balancer feedback-loop handling. The balancer feedback handling is a heuristic that may miss complex flow patterns.

#### `belt_structural.rs` (1,493 lines)
- Entity overlaps, belt throughput, output belt coverage, dead ends, loops, item isolation, inserter conflicts, lane throughput
- **Gap**: `check_belt_throughput` uses per-lane capacity without accounting for inserter limitations. A belt can have sufficient lane capacity but be bottlenecked by inserter pickup rates.

#### `fluids.rs` (854 lines)
- Pipe isolation and fluid port connectivity
- **Gap**: Fluid port mirroring (Space Age feature) is only honored by Factorio 2.0+. The layout engine generates `mirror: true` but doesn't validate that the target Factorio version supports it.

#### `inserters.rs` (583 lines)
- Inserter chain validation and direction checks
- **Gap**: No validation of inserter timing vs. belt speed. An inserter on a fast belt may not complete its swing before the next item arrives.

#### `power.rs` (495 lines)
- Power coverage and pole network connectivity
- **Gap**: `place_poles` creates one horizontal pole line per machine row. Non-adjacent rows may end up disconnected. The `repair_pole_connectivity` pass in `layout.rs` attempts to fix this but uses a simple midpoint heuristic.

### 2.5 Blueprint Export (`blueprint.rs`, ~100 lines) - Solid

- Direct JSON + zlib + base64 encoding (no draftsman dependency)
- Handles entity positioning, direction, recipe, io_type, mirror
- **Gap**: No round-trip validation (export -> parse -> compare). The `blueprint_parser` module exists but isn't integrated into the export pipeline.

### 2.6 WASM Bindings (`wasm-bindings/src/lib.rs`) - Good

- Exposes `solve`, `layout`, `layout_traced`, `layout_streaming`, `export_blueprint`, `validate_layout`, `solve_fixture`, `improve_region_streaming`
- Streaming trace events for live debug overlays
- **Gap**: The WASM API exposes `LayoutResult` directly, which includes the full entity list. For very large layouts (tier 5+), this may cause memory pressure in the browser. Consider streaming or pagination.

### 2.7 Web UI (`web/src/main.ts`, 1,230 lines) - Growing Complex

- PixiJS v8 + pixi-viewport rendering
- Trace overlay, validation overlay, SAT zone overlay, junction zone overlay, ghost tiles overlay
- SAT zone editor (in-canvas painting + SAT completion overlay)
- Streaming renderer with phase animation
- Timeline scrubber for trace event playback
- **Gap**: The main.ts file is 1,230 lines and growing. Module initialization, event handling, and overlay management are all in one file. Consider splitting into smaller modules.

## 3. Technical Debt

### 3.1 Medium Priority

1. **Missing balancer templates** (GitHub #136)
   - The (5,7) balancer template is missing, so copper-cable feeder paths don't get A* specs
   - Impact: Dead-end output belts that do not appear as RouteFailure events
   - Fix: Regenerate balancer library with `python scripts/generate_balancer_library.py`

2. **Fluid row 3-tile pitch** (GitHub #68)
   - Fluid lanes reserve columns but the pitch calculation may not account for all machine orientations
   - Impact: Potential pipe routing conflicts on complex layouts
   - Fix: Audit `ghost_router` step 1 fluid lane reservation logic

3. **Balancer templates oversized** (GitHub #135)
   - Pre-generated balancer templates may be larger than necessary
   - Impact: Layouts are wider than needed
   - Fix: Optimize template generation or add size-aware template selection

4. **No recipe efficiency ranking**
   - The solver picks the first matching recipe without comparing throughput/energy
   - Impact: Suboptimal machine counts for complex recipes
   - Fix: Add recipe scoring and multi-recipe exploration

### 3.2 Low Priority

5. **Hardcoded negotiation parameters**
   - `MAX_NEGOTIATION_ITERATIONS = 8`, `MAX_NO_IMPROVEMENT = 2`, `TURN_PENALTY = 8`
   - Impact: May be insufficient for very dense layouts
   - Fix: Make these configurable or add adaptive tuning

6. **Pole connectivity heuristic**
   - `repair_pole_connectivity` uses a simple midpoint search with 20 iterations
   - Impact: May fail on very wide layouts
   - Fix: Add strategic pole placement or use A* to find optimal bridge locations

7. **No export round-trip validation**
   - Exported blueprints are not automatically parsed and compared
   - Impact: Silent data loss in export (e.g., mirror flag, recipe field)
   - Fix: Add `export -> parse -> diff` test for each e2e test

8. **Web UI monolithic main.ts**
   - 1,230 lines in a single file
   - Impact: Hard to maintain, hard to test
   - Fix: Split into smaller modules (e.g., `main/layout.ts`, `main/overlays.ts`)

## 4. Missing Features / Gaps

### 4.1 Tier 5+ Recipes (GitHub #65, #68)

| Tier | Recipe | Blocker |
|------|--------|---------|
| 4 | `advanced-circuit` from ores | #68 fluid row pitch + #136 missing balancer |
| 5 | `processing-unit` | Deep chain, multiple fluids, no routing failures on small rates |
| 6 | `rocket-control-unit` | Very deep chain |

**Recommendation**: Fix tier 4 from ores first (fluid row pitch + balancer templates), then use the same pipeline for tier 5 with `skip_validation` (as the diagnostic tests already do).

### 4.2 Spaghetti Routing Revival

The `rfp-belt-flow-aware-astar.md` proposal describes a future spaghetti-routing revival. This is an aspirational design that has not been implemented. The ghost pipeline is the current path, but spaghetti routing would be valuable for:
- Non-bus layouts (e.g., compact factories)
- Layouts with complex interdependencies
- Community blueprint analysis and improvement

### 4.3 Multi-Fluid Row Support

The `rfp-multi-fluid-rows.md` proposal has been implemented (`c3a9cbb`), but the implementation may not handle all edge cases. The fluid lane reservation logic in `ghost_router` step 1 needs to be tested against complex multi-fluid layouts.

### 4.4 SAT Zone Editor Polish

The SAT zone editor (Phase F) is implemented but needs:
- Tier-2 validation (belt flow awareness)
- Better error messages for invalid paints
- Export/import of zone configurations

## 7. Conclusion

Fucktorio is a **well-engineered, production-quality** project. The ghost pipeline is sophisticated, the validation framework is comprehensive, and the documentation is exceptional. The main areas for improvement are:

1. **Balancer template coverage** (#136) - a concrete, fixable gap
2. **Fluid row pitch** (#68) - a routing correctness issue
3. **Recipe efficiency ranking** - a solver optimization
4. **Web UI modularization** - a maintainability issue
5. **Tier 5+ recipe support** - a roadmap item

The project is in good shape for continued development. The existing test infrastructure (e2e tests, snapshot debugger, diagnostic tests) provides a solid foundation for future work.
