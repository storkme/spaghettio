# Web App: Rust WASM + PixiJS Interactive Tool

## Vision

Build an interactive web app for the Factorio blueprint generator. Three goals:

1. **Generate blueprints in the browser** — solver → layout → export, all running client-side via WASM
2. **Visualize placement algorithms** — step-by-step replay of how the bus placer, router, and A* work (great for debugging too)
3. **Interactive building** — eventually let users place machines, drag belts, with the solver/router as assistive tools

## Architecture

```
Browser
├── Vanilla TS UI (sidebar: item picker, rate input, solver summary)
├── PixiJS v8 canvas (tile grid with pan/zoom via pixi-viewport)
└── WASM module (spaghettio_core compiled via wasm-bindgen)

Rust Cargo Workspace
├── crates/core/           — shared pure logic (models, solver, recipe DB, blueprint export)
├── crates/pyo3-bindings/  — existing PyO3 A* router (unchanged, used by Python pipeline)
└── crates/wasm-bindings/  — wasm-bindgen thin wrapper over core

Web App (web/)
├── Vite + vanilla TypeScript (no framework)
├── PixiJS v8 + pixi-viewport for the interactive canvas
├── vite-plugin-wasm for WASM imports
└── WASM pkg generated by wasm-pack into web/src/wasm-pkg/
```

## Technology choices

- **Rust/WASM** for the engine (solver, layout, validation, blueprint export)
- **PixiJS v8** for the tile grid renderer (pan/zoom, sprites, hit testing, layers)
- **pixi-viewport** for pan/zoom (eliminates huge chunk of fiddly work)
- **Vanilla TypeScript** for UI chrome (sidebar, controls) — no framework needed for skeleton
- **Vite** for dev server and bundling (handles WASM natively)
- **wasm-pack** to build the WASM module (target: web)

## Status

**MVP shipped (2026-04-05).** The full pipeline runs in the browser: solver → bus layout → blueprint export → validation. The Rust port to `crates/core` is complete (see `docs/archive/port-status.md`). The web app at `web/` is the primary user-facing tool.

### What's live

- WASM module exposes `solve`, `layout`, `export_blueprint`, `validate`, and recipe lookups
- Searchable item picker with machine type selection
- Live solve on parameter change, URL state persistence
- PixiJS canvas with entity rendering, grid, and production graph DAG visualization
- Blueprint string export (copy to clipboard)
- Solver summary with totals in sidebar

## Draftsman dependency analysis

Draftsman is only used for:
1. **Recipe/entity data** (`recipe_db.py`) → replaced by extracted JSON
2. **Fluid port positions** (`validate.py:210-241`) → included in extracted JSON
3. **Blueprint export** (`blueprint/blueprint.py`) → trivially reimplementable (JSON + zlib + base64)
4. **Blueprint import/visualization** (`visualize.py`, `verify.py`, `analysis/`) → not needed for web app

Everything else (solver, layout engines, validation logic) is pure Python with no draftsman deps.
